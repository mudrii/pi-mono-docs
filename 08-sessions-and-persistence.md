# Sessions and Persistence

Pi persists conversations as append-only JSONL files organized as tree structures. This document covers the session format, tree navigation, compaction, settings management, and model resolution.

## JSONL Tree Structure

### File Format

Each session is a single `.jsonl` file where every line is a JSON object. The first line is always the **session header**, followed by **session entries**.

```
{"type":"session","version":3,"id":"abc123","timestamp":"2025-01-15T10:00:00.000Z","cwd":"/home/user/project"}
{"type":"message","id":"e1a2b3c4","parentId":null,"timestamp":"2025-01-15T10:00:01.000Z","message":{...}}
{"type":"message","id":"f5d6e7f8","parentId":"e1a2b3c4","timestamp":"2025-01-15T10:00:05.000Z","message":{...}}
```

### Session Header

```ts
interface SessionHeader {
  type: "session";
  version?: number;      // Current: 3 (v1 sessions lack this)
  id: string;            // UUID
  timestamp: string;     // ISO 8601
  cwd: string;           // Working directory where session was started
  parentSession?: string; // Path to parent session (if forked)
}
```

### Entry Types

All entries share a base structure:

```ts
interface SessionEntryBase {
  type: string;
  id: string;          // 8-char hex, collision-checked
  parentId: string | null; // null for root entry
  timestamp: string;
}
```

The `id`/`parentId` fields form a tree. Each entry is a child of its `parentId`. The full set of entry types:

| Type | Purpose | Participates in LLM context |
|------|---------|---------------------------|
| `message` | User, assistant, or tool result messages | Yes |
| `custom_message` | Extension-injected messages | Yes (converted to user message) |
| `compaction` | Context compaction summary | Yes (as compaction summary) |
| `branch_summary` | Summary of an abandoned branch | Yes (as branch summary) |
| `thinking_level_change` | Records thinking level changes | No (restored on session load) |
| `model_change` | Records model switches | No (restored on session load) |
| `custom` | Extension state persistence | No |
| `label` | User-defined bookmark on an entry | No |
| `session_info` | Session metadata (display name) | No |

### Message Entry

Contains a full `AgentMessage` (from `pi-agent-core`):

```ts
interface SessionMessageEntry extends SessionEntryBase {
  type: "message";
  message: AgentMessage; // user, assistant, or toolResult
}
```

### Custom Message Entry

Extension-injected messages that participate in LLM context:

```ts
interface CustomMessageEntry<T = unknown> extends SessionEntryBase {
  type: "custom_message";
  customType: string;    // Extension identifier
  content: string | (TextContent | ImageContent)[];
  details?: T;           // Extension metadata (not sent to LLM)
  display: boolean;      // true = show in TUI, false = hidden
}
```

### Compaction Entry

Records a context compaction event:

```ts
interface CompactionEntry<T = unknown> extends SessionEntryBase {
  type: "compaction";
  summary: string;
  firstKeptEntryId: string; // ID of first entry kept after compaction
  tokensBefore: number;
  details?: T;              // File tracking data
  fromHook?: boolean;       // true if generated by extension
}
```

### Branch Summary Entry

Records a summary of a branch that was navigated away from:

```ts
interface BranchSummaryEntry<T = unknown> extends SessionEntryBase {
  type: "branch_summary";
  fromId: string;      // Entry ID where the branch diverged
  summary: string;
  details?: T;
  fromHook?: boolean;
}
```

### Custom Entry (Extension State)

For extensions to persist state across session reloads. Not sent to LLM:

```ts
interface CustomEntry<T = unknown> extends SessionEntryBase {
  type: "custom";
  customType: string;
  data?: T;
}
```

### Label Entry

User-defined bookmarks on entries:

```ts
interface LabelEntry extends SessionEntryBase {
  type: "label";
  targetId: string;
  label: string | undefined; // undefined clears the label
}
```

## Session Branching

### How Branching Works

The session is an **append-only tree**. No entries are ever modified or deleted. Branching works by moving the **leaf pointer** to an earlier entry:

1. The `leafId` tracks the current position in the tree
2. `appendMessage()` creates a new entry whose `parentId` is the current `leafId`, then advances `leafId` to the new entry
3. `branch(entryId)` moves `leafId` to an existing entry without creating anything
4. The next `append*()` call creates a child of that entry, forming a new branch

```
Entry A (root)
  Entry B (assistant response)
    Entry C (user follow-up)       <-- original branch
    Entry D (user re-edit)         <-- new branch after branch("B")
      Entry E (assistant response)
```

### Branch with Summary

When navigating away from a branch via `/tree`, pi optionally generates a **branch summary** -- an LLM-generated summary of the abandoned conversation path. This is stored as a `branch_summary` entry that becomes part of the new branch's context:

```ts
branchWithSummary(branchFromId: string | null, summary: string, details?: unknown): string
```

### Fork (New Session File)

Forking creates a **new session file** containing only the path from root to a specified leaf. The new session's header includes a `parentSession` field pointing to the original file:

```ts
createBranchedSession(leafId: string): string | undefined
```

This is triggered by the `/fork` command or `ctx.fork(entryId)` in extensions.

### Cross-project Fork

`SessionManager.forkFrom()` creates a new session in a different project directory with the full history from a source session. The header's `cwd` is updated to the target directory.

## Active Path Tracking (leafId)

The `leafId` is the central concept for navigating the session tree:

- **null**: Before any entries exist (empty session). Next append creates a root entry (`parentId: null`)
- **string**: Points to the current leaf entry. Next append creates a child of this entry

### Context Building: Walking from Leaf to Root

`buildSessionContext()` walks from the leaf entry to the root via `parentId` pointers, collecting the path in order. It then processes entries along this path:

1. **Settings extraction**: Collects `thinking_level_change` and `model_change` entries to determine the current model and thinking level
2. **Compaction handling**: If a compaction entry is found on the path:
   - Emits the compaction summary as the first message
   - Emits "kept messages" (entries between `firstKeptEntryId` and the compaction entry)
   - Emits all messages after the compaction entry
3. **Message conversion**: `message`, `custom_message`, and `branch_summary` entries are converted to `AgentMessage` objects

The result is a `SessionContext`:

```ts
interface SessionContext {
  messages: AgentMessage[];
  thinkingLevel: string;
  model: { provider: string; modelId: string } | null;
}
```

### Tree Visualization

`getTree()` returns the full session as a tree of `SessionTreeNode` objects:

```ts
interface SessionTreeNode {
  entry: SessionEntry;
  children: SessionTreeNode[];
  label?: string; // Resolved label, if any
}
```

Children are sorted by timestamp (oldest first). The `/tree` command renders this tree for navigation.

## Automatic Compaction

### When Compaction Triggers

Compaction triggers automatically when context tokens exceed the model's context window minus a reserve:

```ts
function shouldCompact(contextTokens: number, contextWindow: number, settings: CompactionSettings): boolean {
  if (!settings.enabled) return false;
  return contextTokens > contextWindow - settings.reserveTokens;
}
```

It also triggers on **context overflow** errors from the provider (as a recovery mechanism).

### Compaction Settings

```ts
const DEFAULT_COMPACTION_SETTINGS = {
  enabled: true,
  reserveTokens: 16384,   // Buffer for system prompt + new content
  keepRecentTokens: 20000, // Tokens to keep in recent context
};
```

### Compaction Algorithm

1. **Preparation** (`prepareCompaction()`):
   - Walks the current branch from the last compaction (or start) to the leaf
   - Estimates total context tokens using the last assistant's `usage` field plus trailing message estimates
   - Finds a **cut point** by walking backwards from the newest entry, accumulating token estimates until `keepRecentTokens` is reached
   - Valid cut points are user, assistant, custom, or bashExecution messages (never tool results)
   - If the cut falls mid-turn (on an assistant message, not the turn's user message), records a **split turn** for separate summarization

2. **Summary Generation** (`compact()`):
   - For split turns, generates both a history summary and a turn prefix summary in parallel
   - For normal cuts, generates a single history summary
   - Uses the LLM with a structured prompt that produces Goal/Progress/Key Decisions/Next Steps sections
   - If a previous compaction summary exists, uses an **update prompt** that merges new information into the existing summary
   - Appends file operation tracking (read/modified files) to the summary

3. **Persistence**:
   - Appends a `CompactionEntry` as a child of the current leaf
   - The entry records `summary`, `firstKeptEntryId`, `tokensBefore`, and file tracking `details`

### Token Estimation

Token counts are estimated using a `chars / 4` heuristic. For messages with known usage (from assistant responses), the actual `totalTokens` is used, with trailing messages estimated:

```ts
function estimateContextTokens(messages: AgentMessage[]): ContextUsageEstimate {
  // Uses last assistant's usage.totalTokens + estimates for messages after it
}
```

### Extension Hook: `session_before_compact`

Extensions can intercept compaction via the `session_before_compact` event:

```ts
pi.on("session_before_compact", async (event, ctx) => {
  // event.preparation contains CompactionPreparation
  // event.branchEntries contains the entries being compacted

  // Cancel compaction:
  return { cancel: true };

  // Or provide custom compaction result:
  return {
    compaction: {
      summary: "my custom summary",
      firstKeptEntryId: event.preparation.firstKeptEntryId,
      tokensBefore: event.preparation.tokensBefore,
      details: { myData: true },
    }
  };
});
```

## Branch Summarization

When navigating to a different point in the session tree (via `/tree` or `ctx.navigateTree()`), pi can generate a summary of the branch being left.

### Entry Collection

`collectEntriesForBranchSummary()` walks from the old leaf back to the **common ancestor** with the target, collecting all entries along the way. It does not stop at compaction boundaries -- their summaries become context.

### Summary Generation

`generateBranchSummary()`:
1. Prepares entries with a token budget (`contextWindow - reserveTokens`)
2. Walks entries newest-to-oldest, fitting messages within budget
3. Extracts file operations from tool calls and existing branch summary details
4. Serializes the conversation to text and calls the LLM with a structured prompt
5. Prepends a preamble: "The user explored a different conversation branch before returning here."
6. Appends file operation tracking

The result is stored as a `branch_summary` entry at the branch point.

### Branch Summary Settings

```ts
interface BranchSummarySettings {
  reserveTokens?: number; // default: 16384
  skipPrompt?: boolean;   // default: false - skip "Summarize branch?" prompt
}
```

## Settings Management

### SettingsManager

The `SettingsManager` handles global and project-scoped settings with automatic persistence.

> **Removed in v0.65.0:** The `session_directory` configuration key has been removed from both the extension API and the settings API. Extensions and SDK users that previously depended on it should use `sessionManager.getSessionFile()` to derive the session location, then call `path.dirname()` to get the containing directory.

### Storage Locations

- **Global**: `~/.pi/agent/settings.json`
- **Project**: `<cwd>/.pi/settings.json`

Project settings override global settings. Nested objects are merged recursively (not replaced). Arrays and primitives use project value when present.

### File Locking

Settings files use `proper-lockfile` for safe concurrent access. The `withLock()` method acquires a lock, reads the current content, applies changes, and writes back. Lock acquisition retries up to 10 times with 20ms delay.

### Write Queueing

Settings writes are queued to avoid concurrent file operations. `SettingsManager.flush()` waits for all pending writes.

### Migration

Old settings formats are automatically migrated:
- `queueMode` renamed to `steeringMode`
- `websockets` boolean migrated to `transport` enum
- `skills` object format migrated to array format

### In-Memory Mode

`SettingsManager.inMemory()` creates a settings manager with no file I/O, useful for testing and the SDK.

## Model Resolution

### ModelRegistry

The `ModelRegistry` manages all available models:

1. **Built-in models**: from `pi-ai` package (Anthropic, OpenAI, Google, etc.)
2. **Custom models**: from `~/.pi/agent/models.json`
3. **Extension-registered models**: from `pi.registerProvider()`

Custom models in `models.json` can override built-in models (same provider + ID replaces). OAuth providers can modify models after login (e.g., updating base URLs).

### API Key Resolution

API keys are resolved through `AuthStorage` with a fallback chain:
1. OAuth credentials (if provider uses OAuth)
2. Environment variables (standard per-provider)
3. Custom provider API key from `models.json` (supports `$ENV_VAR` syntax and shell command resolution)

### Model Resolution (Fuzzy Matching)

The `resolveCliModel()` function handles flexible model specification:

1. **Canonical match**: `provider/modelId` (exact)
2. **Bare ID match**: `modelId` (must be unambiguous across providers)
3. **Partial match**: substring of model ID or name
4. **Alias preference**: When multiple models match, aliases (without date suffix) are preferred over dated versions
5. **Latest dated**: Among dated versions, the latest (highest-sorting ID) wins

Model patterns can include a thinking level suffix: `claude-sonnet:high`. The algorithm tries the full string first, then progressively strips colon-suffixes to find a match.

### Model Scoping (`--models` flag)

The `resolveModelScope()` function resolves patterns from the `--models` CLI flag or settings:

- Supports glob patterns: `anthropic/*sonnet*`, `*opus*`
- Each pattern can include a thinking level: `claude-sonnet:high`
- Matched models are deduplicated
- Used for Ctrl+P model cycling

### Initial Model Selection

`findInitialModel()` follows this priority:

1. CLI args (`--provider` + `--model`)
2. First scoped model (from `--models`, if not continuing/resuming)
3. Restored from session (if continuing/resuming)
4. Saved default from settings (`defaultProvider`/`defaultModel`)
5. First available model with valid API key (tries known defaults first)

### Default Models per Provider

Each known provider has a default model ID:

| Provider | Default Model |
|----------|--------------|
| anthropic | claude-opus-4-6 |
| openai | gpt-5.4 |
| google | gemini-2.5-pro |
| xai | grok-4-fast-non-reasoning |
| openrouter | openai/gpt-5.1-codex |
| kimi-coding | kimi-k2-thinking |

(See `model-resolver.ts` for the complete list.)

## Session Selector and Navigation

### Session Listing

`SessionManager.list()` lists all sessions for a directory. `SessionManager.listAll()` lists across all project directories under `~/.pi/agent/sessions/`.

Each session is represented as `SessionInfo`:

```ts
interface SessionInfo {
  path: string;
  id: string;
  cwd: string;              // Working directory
  name?: string;            // User-defined display name
  parentSessionPath?: string;
  created: Date;
  modified: Date;           // Last activity time
  messageCount: number;
  firstMessage: string;     // First user message text
  allMessagesText: string;  // All messages concatenated for search
}
```

### Session Directory Encoding

Session files are stored in `~/.pi/agent/sessions/<encoded-cwd>/`. The cwd is encoded by replacing path separators with hyphens and wrapping in `--`:

```
/home/user/project  ->  --home-user-project--/
```

### Resource Precedence (v0.65.1+)

Project-level and user-level resources (skills, prompt templates, themes) now consistently override package-bundled resources. CLI-provided paths (via `-e`, `--skill`, `--prompt-template`, `--theme`) take the highest precedence over all discovered resources.

Previously, package resources could incorrectly override project/user resources in some discovery paths.

### Session Persistence Behavior

Sessions use **deferred persistence**: the file is not written until the first assistant response arrives. This avoids creating session files for sessions where the user never sends a message.

The `_persist()` method:
1. Checks if any assistant message exists in the entries
2. If no assistant message yet, marks the session as "not flushed" and defers writing
3. On the first assistant response, writes all accumulated entries at once
4. After that, each new entry is appended individually

### Session Creation Methods

| Method | Behavior |
|--------|----------|
| `SessionManager.create(cwd)` | New session in default directory |
| `SessionManager.open(path)` | Open specific session file |
| `SessionManager.continueRecent(cwd)` | Most recent session or new |
| `SessionManager.inMemory()` | No file persistence |
| `SessionManager.forkFrom(source, cwd)` | Fork from another project |

## AgentSessionRuntime (v0.65.0+)

For SDK users who need to switch between sessions at runtime, `AgentSessionRuntime` wraps session lifecycle with a `CreateAgentSessionRuntimeFactory` closure. The factory recreates all cwd-bound services on every session switch, ensuring `/new`, `/resume`, `/fork`, and import all follow identical initialization.

See `packages/coding-agent/docs/sdk.md` and `examples/sdk/13-session-runtime.ts` for the full API reference and working example.

Key operations:

```typescript
await runtime.newSession();
await runtime.switchSession("/path/to/session.jsonl");
await runtime.fork("entry-id");
// After each: runtime.session is the new live session
```

### Missing Session Working Directory (v0.65.1+)

When resuming or importing a session whose original working directory no longer exists:
- **Interactive mode**: Prompts the user whether to continue in the current working directory.
- **Non-interactive mode**: Fails with a clear, descriptive error message.

Previously this case could produce cryptic errors or unexpected behavior.

## Migration System

### Version History

- **v1**: Linear entries without id/parentId (original format)
- **v2**: Tree structure with id/parentId, compaction uses `firstKeptEntryId` instead of index
- **v3**: Renamed `hookMessage` role to `custom`

### Migration Process

`migrateToCurrentVersion()` runs all needed migrations in sequence, mutating entries in place:

- **v1 to v2**: Generates 8-char hex IDs for all entries, sets `parentId` to previous entry's ID (linear chain), converts `firstKeptEntryIndex` to `firstKeptEntryId` in compaction entries
- **v2 to v3**: Updates message entries with `hookMessage` role to `custom`

After migration, the file is rewritten with `_rewriteFile()`.

### Compatibility

Old session files are automatically migrated on load. The migration is transparent -- the user sees no difference. The session version is stored in the header and checked on every load.

## Built-in Tools

Pi provides seven built-in tools:

| Tool | Description |
|------|-------------|
| `read` | Read file contents with offset/limit support |
| `bash` | Execute shell commands |
| `edit` | Edit files (search-and-replace) |
| `write` | Write/create files |
| `grep` | Search file contents with regex |
| `find` | Find files by name pattern |
| `ls` | List directory contents |

### Default Active Tools

The default active set is `[read, bash, edit, write]`. Additional tools (`grep`, `find`, `ls`) are available but not active by default.

### Tool Creation

Tools are created via factory functions that bind to a working directory:

```ts
const tools = createAllTools(cwd, {
  read: { /* ReadToolOptions */ },
  bash: { /* BashToolOptions */ },
});
```

### Tool Override by Extensions

Extensions can register tools with the same name as built-in tools. The extension tool replaces the built-in. Extension tools are wrapped via `wrapRegisteredTool()` to inject the `ExtensionContext` as the final argument to `execute()`.

### Tool Interception

All tool calls pass through the agent's `beforeToolCall` and `afterToolCall` hooks, which delegate to extension `tool_call` and `tool_result` events respectively. This allows extensions to:
- Block tool execution (return `{ block: true, reason: "..." }`)
- Modify tool results (return `{ content: [...], details: {...} }`)
- Log or audit tool usage

Tools execute in parallel by default (the agent runs all tool calls from an assistant message concurrently).
